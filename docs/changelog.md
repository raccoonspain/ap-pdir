## 2026-07-13 — Деплой перенесён на VPS rub24 (минуя b24.blackboxbegin.space)
- Первый деплой на `b24.blackboxbegin.space` не завершился: Caddyfile там
  требует root для правки, install-flow не мог доехать до `index.php`
- По решению остановиться и повторить схему `ap-pp` (см. D-009 в его
  decisions.md): `deploy.sh` переписан на SSH-деплой на
  `deploy@45.91.55.178:/var/www/b24/ap-pdir/` (rub24.blackboxbegin.space)
- На rub24: сгенерирован `env.php`, вписаны `B24_CLIENT_ID`/`SECRET`,
  добавлен `handle /ap-pdir*` в Caddyfile (там он без root, правится по
  SSH напрямую), reload — маршрут отвечает штатным 403-guard'ом
- `CLAUDE.md` и `how-to-link.md` обновлены под новый сервер; локальный
  артефакт `/var/www/b24/ap-pdir` на исходной машине удалён (недостижим)
- Зачем: rub24 — целевой сервер проекта (ниже задержка до портала,
  проще эксплуатация Caddy); раз `ap-pdir` ещё не был живым, дешевле
  сразу выбрать конечный сервер, чем сначала поднимать на старом
- Осталось: обновить handler/install URL в карточке local-app в Б24 на
  `rub24.blackboxbegin.space` и повторно открыть приложение для install-flow
- Связанные решения: D-005

## 2026-07-04 — env.php генерируется deploy.sh, init.php — fallback для 1%-кейса
- `deploy.sh` при первом деплое slug'а сам создаёт `env.php` из `env.example`
  с подставленными `APP_URL`/`APP_PATH`/`DATA_ROOT` (детерминированы конвентом
  VPS+Caddy); человеку остаётся вписать только `B24_CLIENT_ID`/`SECRET`
- `init.php` оставлен в шаблоне без изменений, но выведен из основного flow —
  теперь это инструмент для хостинга, где путь/домен заранее не известны
- Новый файл `how-to-link-ALL.md` — процедура привязки к Б24 для этого
  1%-кейса (через `init.php`, с ручной правкой `APP_URL`, если нужно)
- `how-to-link.md`, `README.md`, `CLAUDE.md`, `docs/state.md`, `docs/onboarding.md`
  обновлены под новый (99%-кейс) flow
- Зачем: задокументированный flow `deploy.sh → открыть init.php` не
  воспроизводим как описано — `deploy.sh` исключает `init.php` из rsync,
  на свежем сервере файла там просто нет. На практике оба живых проекта
  на VPS обошли это, написав `env.php` вручную — то есть шаг был лишним
  и вдобавок нерабочим
- Связанные решения: D-003
