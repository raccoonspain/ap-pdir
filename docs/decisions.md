# Журнал решений

> Здесь — **почему** проект устроен так, а не иначе. Каждое значимое
> решение: что выбрали, какие были варианты, почему именно это.
> Файл **только дописывается** (новые записи — вниз). Старые записи не
> удаляем: если решение отменили — добавляем новую запись «отменяет D-NNN».

> Зачем: чтобы следующий исполнитель не «переоткрывал» уже закрытые вопросы
> и не ломал то, что сделано осознанно. Один абзац здесь экономит часы.

Записывай решение, когда выбор был **неочевидным** или у него есть
последствия: скоупы, структура данных, плейсменты, обход бага и т.п.

## D-001 · Шаблон _b24-single-php (single-tenant PHP)
**Дата:** — · **Статус:** принято

**Контекст.** Стартовали новый проект для Битрикс24.
**Решение.** PHP-бэкенд, файловый store (`<?php exit;?>`-защита), vanilla JS, Netangels.
**Почему.** Готовый скелет с OAuth, install-flow, session-gate «из коробки».
Без БД, без composer — минимум зависимостей для single-tenant.
**Последствия.** Multi-tenant / маркет — смена шаблона, не доработка этого.

## D-002 · Переезд хостинга на VPS + Caddy + PHP-FPM
**Дата:** 2026-07-03 · **Статус:** отменяет хостинг-часть D-001

**Контекст.** D-001 фиксировал хостинг как Netangels (shared, `.htaccess`).
Команда перешла на собственный VPS с Caddy + PHP-FPM для всех B24-приложений.

**Варианты.** Остаться на shared-хостинге (Netangels) vs собственный VPS с
явной настройкой каждого проекта через Caddyfile.

**Решение.** VPS, домен `b24.blackboxbegin.space`, один Caddy-блок `handle
/<slug>*` на проект, деплой — локальный rsync (`bash deploy.sh`) прямо на
сервере, без SSH.

**Почему.** Больше контроля над PHP-FPM/Caddy-конфигом, не завязаны на
ограничения конкретного shared-хостера. Caddy не читает `.htaccess` —
защита state осталась той же (`<?php exit;?>`-префикс), это не пришлось
менять.

**Последствия.** `.htaccess`-независимость шаблона (D-001) оказалась кстати:
переезд не потребовал менять модель защиты. Появились новые следствия
Caddy `uri strip_prefix` — APP_URL нужно править вручную после `init.php`,
handler URL обязательно с `/index.php`, `template.html` вместо `index.html`.
Все зафиксированы в `CLAUDE.md` → «Caddy + PHP-FPM — следствия для шаблона».
Домен, а не IP — при смене хостера под тем же доменом менять ничего не
нужно; деплой на сервер заказчика с другим доменом — отдельный кейс,
не покрыт этим решением.

## D-003 · env.php генерируется deploy.sh, init.php — только для 1%-кейса
**Дата:** 2026-07-04 · **Статус:** принято, уточняет D-002

**Контекст.** После D-002 конвент деплоя зафиксирован: домен всегда
`b24.blackboxbegin.space`, путь всегда `/var/www/b24/<slug>`. При этом
`APP_URL`/`APP_PATH`/`DATA_ROOT` стали полностью детерминированы из
одного значения (slug), но флоу продолжал требовать открыть `init.php`
в браузере и вручную поправить `APP_URL` (следствие `uri strip_prefix`
из D-002). На практике оба реальных проекта на VPS (`ap-simple-test`,
`fv-taskindep-panel`) этот шаг обошли — `env.php` был написан вручную
напрямую, `init.php` на сервере не оказывалось вообще (`deploy.sh` его
и не копирует, см. `--exclude 'init.php'`), задокументированный флоу
не воспроизводим на свежем сервере как описано.

**Варианты.** (а) Почистить `deploy.sh`, чтобы `init.php` таки доезжал
до сервера при первом деплое — оставить flow как есть. (б) Убрать
`init.php` из основного flow, генерировать `env.php` детерминированно
прямо в `deploy.sh`, оставить `init.php` в шаблоне только как fallback
для хостинга, где путь/домен заранее не известны (shared-хостинг
заказчика, FTP).

**Решение.** (б). `deploy.sh` при первом деплое slug'а (env.php на
сервере ещё нет) генерирует `env.php` из `env.example` подстановкой
`APP_URL=https://b24.blackboxbegin.space/<slug>`, `APP_PATH=DEPLOY_DIR`,
`DATA_ROOT=DEPLOY_DIR/data`. Человеку остаётся вписать только
`B24_CLIENT_ID`/`SECRET` — их и так нельзя получить иначе, кроме как
из карточки local-app в Б24. `init.php` физически остаётся в шаблоне
(`www/init.php`) как есть, без изменений — для кейса, где конвент
VPS+Caddy не применим (хостинг заказчика с непредсказуемым путём).

**Почему.** Автоопределение `APP_URL` через `init.php` было нужно для
общего случая (произвольный shared-хостинг, куда шаблон изначально
целился) — на своём VPS с фиксированным конвентом это лишний шаг,
который к тому же не работал как описано (см. «Контекст»). Убирать
`init.php` из репозитория целиком не стали — шаблон должен оставаться
пригодным не только для нашего VPS.

**Последствия.** `how-to-link.md` описывает только 99%-кейс (наш VPS,
`deploy.sh`). Новый файл `how-to-link-ALL.md` — процедура для 1%-кейса
через `init.php`, на него ссылаются `how-to-link.md`, `README.md`,
`CLAUDE.md`. `docs/state.md`-чеклист (шаблонный, наследуется новыми
проектами) обновлён под новый flow.

## D-004 · Макет пульта — вариант B (таблица + KPI-сводка), без кэша на первой итерации
**Дата:** 2026-07-13 · **Статус:** принято

**Контекст.** Нужно было выбрать макет главной страницы пульта и способ
загрузки данных смарт-процессов (Сделка/Этап/Модуль/Оплата, ~4000 модулей
на портале). Сделано три черновика: A — плоская таблица без сводки,
B — таблица + KPI-плашки сверху, C — группировка по стадии.

**Варианты.** Макет: A / B / C (см. выше). Загрузка данных: считать
агрегаты (счётчики по стадиям, индикаторы срыва/оплаты) на клиенте после
раскрытия строки, либо посчитать их на сервере сразу и отдать фронту
готовое дерево. Кэширование: заводить файловый кэш сразу или отложить.

**Решение.** Макет — B (KPI-плашки сверху + таблица сделок + аккордеон
Сделка→Этап→Модуль). Данные — один эндпоинт `www/api/dashboard.php`,
собирает всё через `batch()`-запросы (`crm.item.list` постранично) и
агрегирует на сервере в `fetchDashboardData()` (`api/lib.php`); раскрытие
строки на фронте — чистый рендер уже полученного дерева, без новых REST.
Кэш — не делаем на этой итерации.

**Почему.** B даёт директору сводную оценку (сколько сделок, на какую
сумму, сколько горит) без скролла таблицы — то, ради чего вообще
затевался пульт, а не просто список сделок (A) или разбивка не по тому
измерению, которое важно директору (C, по стадии). Агрегация на сервере
и единый батч-запрос при открытии страницы — раскрытие аккордеона
мгновенное и не плодит REST-вызовы (см.
`rules/rule-b24-rest-batch-not-loop.md`); ожидаемый объём (~4000 модулей)
укладывается в 2-3 batch-запроса. Кэш отложен — на первой итерации важнее
корректность агрегатов, а не задержка одного REST-похода при открытии
страницы; преждевременная оптимизация.

**Последствия.** `fetchDashboardData()` в `api/lib.php` — точка, куда
позже можно добавить файловый кэш с TTL, не трогая остальной код и
фронт. UI и парсинг ответа `dashboard.php` пишутся под контракт «дерево
сделка→этапы→модули с готовыми агрегатами», а не «плоские списки
сущностей» — см. `docs/superpowers/specs/2026-07-13-pult-rukovoditelya-design.md`.

## D-005 · Деплой сразу на VPS rub24, минуя b24.blackboxbegin.space
**Дата:** 2026-07-13 · **Статус:** принято

**Контекст.** Начали разворачивать `ap-pdir` по конвенту шаблона (VPS
`b24.blackboxbegin.space`): задеплоили `bash deploy.sh`, зарегистрировали
local-app в Б24 с handler'ом на этот домен, получили `client_id`/`secret`.
Install-flow не завершился (Caddyfile на этом сервере не был настроен под
`/ap-pdir`, а правка требует root — `deploy`-юзер не может писать в
`/etc/caddy/`). До того как чинить это на старом сервере, пользователь
попросил остановиться: проект переезжает на другой VPS — по образцу
`ap-pp`, который уже прошёл аналогичный переезд на `rub24.blackboxbegin.space`
(`45.91.55.178`, см. D-009 в `ap-pp/docs/decisions.md`) ради меньшей
задержки до портала `alfa-prj.bitrix24.ru`.

**Варианты.** (а) Починить Caddyfile на `b24.blackboxbegin.space` (нужен
root/sudo с паролем — блокер) и остаться на нём. (б) Сразу деплоить на
`rub24.blackboxbegin.space`, повторив паттерн `ap-pp/deploy.sh` (SSH,
ключ `id_ed25519_rub24`, `deploy`-юзер там владеет `/etc/caddy/Caddyfile`
напрямую — правка без sudo).

**Решение.** (б). `ap-pdir` ещё не был живым (ни одной успешной установки,
токены не сохранились) — в отличие от `ap-pp` это не «cutover» с живого
прод-инстанса, а выбор целевого сервера с нуля. `deploy.sh` переписан по
образцу `ap-pp/deploy.sh` (SSH вместо локального rsync). Local-app в Б24
переиспользован — тот же `client_id`/`secret`, но handler/install path
пере-регистрируются на новый домен.

**Почему.** У `rub24` ниже сетевая задержка до портала (см. измерения в
D-009 проекта `ap-pp`: 77мс вместо 401мс round-trip) и `/etc/caddy/Caddyfile`
там правится без root — меньше трения при добавлении новых проектов.
Раз `ap-pdir` не запущен в проде, нет причины сперва разворачивать на
`b24.blackboxbegin.space` и потом переезжать — дешевле сразу выбрать
целевой сервер.

**Последствия.** `deploy.sh`, `CLAUDE.md` (`Деплой`, `О ПРОЕКТЕ`,
`ИНФРАСТРУКТУРА СЕРВЕРА`) и `how-to-link.md` переписаны под rub24.
Остаток проектов на старом сервере (`ap-simple-test`, `fv-taskindep-panel`,
`ap-kilotest2`) не затронут — это решение только про `ap-pdir`, не про
перенос всего домена (см. вариант (в), отклонённый в D-009 `ap-pp` по той
же причине). Локальный артефакт `/var/www/b24/ap-pdir` (создан этой
сессией на исходном VPS, где крутится сам Клод) удалён — недостижим,
Caddyfile под него так и не был написан.

## D-006 · Фильтр сделок (active/all/closed) — постфильтр в PHP, не REST-filter; фетч Pay добавлен в архитектуру
**Дата:** 2026-07-13 · **Статус:** принято, уточняет D-004

**Контекст.** При реализации `fetchDashboardData()` (см. спеку и D-004)
обнаружились два расхождения с архитектурой из спеки:
1. Спека определяет 🟡-индикатор через стадию записи **Pay** («Счёт
   отправлен», entityTypeId 1058), но раздел «Архитектура загрузки данных»
   перечисляет только 3 REST-шага (1050/1054/1062) — Pay не упомянут.
2. Реальный REST-ответ `crm.item.list` для `select`/полей — camelCase
   (`stageId`, `ufCrm13OCode`), а не классические коды из `/source`
   (`STAGE_ID`, `UF_CRM_13_O_CODE`) — см. новое правило
   `rules/rule-crm-item-camelcase-select.md`. Дальше вопрос: фильтровать
   пресет «Активные/Все/Завершённые» через REST `filter` по `stageId`
   (нужен полный код вида `DT1050_21:NEW` — категория не гарантирована
   без отдельного запроса) или в PHP после фетча.

**Варианты.** (1) Pay: добавить 4-й REST-шаг в архитектуру или убрать
Pay-условие из индикатора. (2) Фильтр пресета: REST `filter` с полным
префиксом стадии (`DT<entityTypeId>_<categoryId>:CODE`) — требует заранее
знать `categoryId` — или пост-фильтр в PHP по суффиксу `stageId` после
`:` (не требует знания `categoryId`, дешёво при реалистичном объёме
~50 сделок).

**Решение.** (1) Добавлен 4-й шаг: `crm.item.list` entityTypeId=1058,
`filter: {parentId1054: [...]}` по id уже загруженных этапов —
выполняется только если этапы вообще нашлись. (2) Пресет фильтруется в
PHP: сделки грузятся из Б24 без REST-фильтра по стадии, `stageCode`
(суффикс после `:`) сравнивается со списком закрытых стадий
(`SUCCESS`/`FAIL`) в `dashboardDealMatchesPreset()`.

**Почему.** (1) Без Pay 🟡-индикатор «Ожидание оплаты» неполный — часть
случаев (счёт выставлен, но этап ещё не в стадии «Оплата») не ловится.
Раз спека явно описывает это правило в разделе «Правила статуса», это
пробел в разделе «Архитектура», а не сознательный вырез — дозаполнено.
(2) Постфильтр в PHP не зависит от `categoryId` (который пришлось бы
получать отдельным запросом `crm.category.list` или хардкодить, рискуя
разойтись при смене категории на портале) и на объёме ~50 сделок не
создаёт заметных накладных расходов — REST-оптимизация по фильтру из
спеки была рассчитана на кейс с большим числом сделок, которого пока нет.

**Проверено.** Реальным REST-вызовом на портале `alfa-prj.bitrix24.ru`:
6 сделок, ручной пересчёт `lagDays`/индикаторов/KPI по сырым данным
Milestone/Module совпал с выводом `fetchDashboardData()` для всех трёх
пресетов (`active`/`all`/`closed`).

**Последствия.** Если объём сделок вырастет на порядок, постфильтр в PHP
стоит пересмотреть в пользу REST-`filter` (уже зная `categoryId` из
реального использования). `www/api/dashboard-data.php` — бизнес-логика
вынесена туда отдельно от generic `www/api/lib.php` (см. комментарий в
шапке `lib.php`), эндпоинт `www/api/dashboard.php` — тонкая обвязка
(session-gate + JSON).

## D-007 · Цвета стадий — сняты с воронки Б24 через crm.status.list, не подобраны на глаз
**Дата:** 2026-07-13 · **Статус:** принято

**Контекст.** После первой проверки UI в браузере пользователь показал
скриншоты воронок Deal/Milestone/Module в настройках Б24 (у каждой
стадии — свой цвет, заданный админом) и попросил применить те же цвета
к бейджам стадий на пульте, а не изобретать свою палитру.

**Решение.** Цвета не сняты «на глаз» со скриншота — получены точным
REST-вызовом `crm.status.list(filter: {ENTITY_ID: "DYNAMIC_<entityTypeId>_STAGE_<categoryId>"})`
для каждой из 4 сущностей (categoryId подтверждён отдельно через
`crm.category.list`: 1050→21, 1054→23, 1062→27, 1058→25) и захардкожены
в `DASHBOARD_DEAL_STAGES`/`DASHBOARD_MILESTONE_STAGES`/`DASHBOARD_MODULE_STAGES`
(`www/api/dashboard-data.php`) рядом с названием и порядком стадии.
Эндпоинт отдаёт `stageColor` вместе с `stageName`, фронт красит бейдж
через inline `style` и сам выбирает чёрный/белый текст по формуле
относительной яркости (`0.299r+0.587g+0.114b`, порог 0.6) — так же,
как визуально решает Б24 (проверено на всех 21 стадии из трёх воронок:
совпадает).

**Почему.** REST — источник истины: цвет админ мог поменять в любой
момент, глазами со скриншота легко ошибиться на пару тонов и разойтись
с реальной воронкой. Формула контраста, а не хардкод «эта стадия светлая,
эта тёмная» — не пришлось бы держать в уме и обновлять список при
добавлении новой стадии с любым цветом.

**Последствия.** Если админ перекрасит стадию в Б24, бейдж на пульте
не обновится сам — цвета захардкожены в PHP-константах (тот же trade-off,
что и у названий стадий, зафиксированных в D-004/D-006). Если это станет
проблемой — можно один раз при старте `fetchDashboardData()` подтягивать
`crm.status.list` вместо констант, ценой лишнего REST-вызова на каждое
открытие пульта. Заодно исправлен смежный CSS-баг, найденный на том же
скриншоте: `.error-banner { display: flex }` перебивал атрибут `hidden`
(одинаковая специфичность, автор-стиль позже UA-стиля в каскаде) — баннер
ошибки был виден пустым поверх успешно загруженных данных; добавлено
`.error-banner[hidden] { display: none }`.

## D-008 · Визуальный язык пульта — избранные элементы bugster.dev, без бренда/маскотов
**Дата:** 2026-07-13 · **Статус:** принято

**Контекст.** Пользователь показал referens-дизайн bugster.dev (лендинг
QA-инструмента, стиль «dev-tool стикербук»: зернистая бумажная текстура,
жирные чёрные обводки карточек с крупным скруглением, моноширинный шрифт
даже в заголовках/кнопках, мелкие квадратные «хендлы» по углам рамок как
в Figma, пастельные пилюли-кнопки, рукописные маскоты-иллюстрации) и
попросил применить этот стиль к пульту руководителя.

**Варианты.** (а) Буквально перенести весь визуальный язык, включая
пастельные заливки кнопок и иллюстрации-маскоты. (б) Взять только
структурные/графические приёмы (текстура, обводки, моношрифт, хендлы),
не трогая цветовую семантику пульта и не добавляя декоративных
иллюстраций. (в) Не менять дизайн вовсе.

**Решение.** (б). Перенесены: зернистая SVG-текстура фона (`feTurbulence`
data-URI) вместо прежнего точечного узора; чёрные обводки (`--ink`) и
увеличенный радиус скругления у KPI-блока и таблицы; KPI-плашки собраны
в одну рамку с разделителями вместо 4 отдельных карточек (паттерн
«10x / <5min / Zero» с исходника); моноширинный шрифт (`--font-mono`) —
заголовок, KPI-числа, шапка таблицы, кнопки-пресеты; квадратные
`.corner-handle` по углам обеих рамок. Не перенесено: пастельная палитра
кнопок/заливок, маскоты и декоративные иллюстрации, полный переход на
моноширинный шрифт (плотный текст ячеек — названия, активность —
остался sans-serif). Цвета плашек стадий (D-007, реальные цвета из
воронки Б24) не тронуты ни на йоту — это не брендинг, а данные.

**Почему.** Пульт — рабочий инструмент для директора и замов, не
маркетинговый лендинг: маскоты и пастельные стикеры создали бы
несерьёзное впечатление и отвлекали бы от чисел. При этом графические
приёмы (текстура вместо плоского белого, жирная обводка вместо мягкой
тени, моношрифт для чисел/лейблов) — стилистически нейтральны и хорошо
ложатся на «control-panel»-природу самого приложения (см. название
«Пульт руководителя»). Кириллица в моноширинном шрифте в длинных строках
(названия сделок, тексты активности) читается хуже, чем в sans-serif —
поэтому моно ограничен короткими элементами (числа, лейблы, код).

**Последствия.** Дальнейшие UI-доработки пульта следует держаться этого
языка (текстура/обводки/моно для чисел и лейблов), а не возвращаться к
плоским белым карточкам с мягкими тенями — если понадобится другое
направление, это отдельное решение, не тихий откат. Цвета стадий и их
источник (`crm.status.list`, D-007) — по-прежнему единственный источник
цветовой семантики, бренд-референсы вроде bugster.dev на них не влияют.
